# BD Coaching Pipeline

## Overview

Backend pipeline that produces BD coaching scores + Indonesian narrative from the SAME Claude transcript call used in `06-claude-extraction-pipeline.md`. The Claude call returns TWO JSON objects in a single response: (a) the extraction object, (b) the coaching object. Backend splits them, persists separately, and routes the coaching narrative to the BD owner via Telegram + email at 08:00 WIB the next morning.

Reuses Stage 2 prompt (`CLAUDE_BANTS_SCORING_PROMPT`) augmented with the SOP rubric below (`CLAUDE_COACHING_RUBRIC` system_config key). No additional Claude API spend — single call, dual output.

Business reference: `context/claude/22-bd-objection-handling.md` (post-call objection patterns), `context/for-business/SOP-BD-Discovery-Call.pdf` (26-question rubric source).

## 26-Question SOP Rubric

Per-question score 0–3. Aggregated by call phase; phase scores feed the 5-dimension Quality Score.

| Phase | # Questions | Sample question (representative) |
|---|---:|---|
| **Opening** | 4 | "Did BD set agenda within first 60s?" |
| **Discovery — Pain** | 6 | "Did BD probe quantified pain (Rp/jam/jumlah)?" |
| **Discovery — Authority** | 4 | "Did BD identify decision-maker by name + title?" |
| **Demo** | 5 | "Was demo tied to client's stated pain (not generic feature tour)?" |
| **Closing** | 4 | "Did BD propose specific next step with date?" |
| **Post-Call** | 3 | "Did BD send recap email within 24h?" |

Total: 26 questions × max 3 points = 78 raw points.

```
phase_score(phase) = sum(question_scores in phase) / (3 * num_questions in phase)
                   → normalised 0.00 – 1.00 per phase
```

Claude returns each question score with a transcript evidence quote.

## 5-Dimension Quality Score

```
overall = sop * 0.30
        + discovery * 0.25
        + feature_gap * 0.20
        + objection * 0.15
        + demo * 0.10
```

| Dimension | Source | Range |
|---|---|---|
| `sop` | mean(all 26 phase scores normalised) | 0.00 – 1.00 |
| `discovery` | mean(Discovery-Pain + Discovery-Authority phase scores) | 0.00 – 1.00 |
| `feature_gap` | inverse of "feature_gap_objections_unhandled / total_objections_raised" | 0.00 – 1.00 |
| `objection` | mean(per-objection handling score, 0–3 per occurrence) / 3 | 0.00 – 1.00 |
| `demo` | mean of Demo-phase question scores normalised | 0.00 – 1.00 |

Final `overall` ∈ `[0.00, 1.00]`. Stored as DECIMAL(3,2). Multiply by 10 for FE display ("7.4 / 10").

## 5-Component Deal Score

```
overall_deal = next_step * 0.30
             + pain_quantified * 0.20
             + dm_identified * 0.20
             + urgency * 0.20
             + product_fit * 0.10
```

| Component | Computation |
|---|---|
| `next_step` | 1.0 if specific date+activity agreed; 0.5 if vague; 0.0 if absent |
| `pain_quantified` | 1.0 if Rp/jam/jumlah cited; 0.5 if qualitative; 0.0 if no pain mentioned |
| `dm_identified` | 1.0 if name+title+presence; 0.5 if name only; 0.0 if absent |
| `urgency` | 1.0 if `<30 days`; 0.7 if `30–90`; 0.4 if `90–180`; 0.0 if `>180` or vague |
| `product_fit` | from extraction `product_match_confidence` field |

`overall_deal` ∈ `[0.00, 1.00]`, stored as DECIMAL(3,2).

## Stall Prediction Integration

Backend cron `stall_predictor` runs at 23:00 WIB daily and queries:

```sql
SELECT prospect_id, workspace_id, bd_owner_email
FROM bd_deal_scores
WHERE call_day = 'D2'
  AND overall_deal < 0.5
  AND prospect_id NOT IN (SELECT prospect_id FROM escalations WHERE escalation_id = 'ESC-BD-003');
```

For each result → schedule `ESC-BD-003` to fire at D7. Escalation flow defined in `context/claude/16-guard-manual-flows-strategy.md`.

## Coaching Narrative Format

250–400 word Indonesian text, structured. Generated by Claude as a single string field in the coaching JSON output. No HTML — markdown-lite (line breaks + bullet dashes only).

```
Struktur call:
[1–2 sentence overview of how the call flowed: opening, discovery time, demo length, close]

Yang berjalan baik:
- [bullet 1 — specific behavior + transcript timestamp/quote]
- [bullet 2]
- [bullet 3]

Yang perlu diperbaiki:
- [bullet 1 — specific behavior + improvement suggestion]
- [bullet 2]
- [bullet 3]

Verdict:
[1 paragraph: deal probability, suggested next-step adjustments, coaching priority for next 1-on-1]
```

Backend MUST validate the narrative contains all 4 section headers verbatim before persisting; else mark as `narrative_status='invalid_format'` and re-prompt Claude with stricter instructions (1 retry).

## Auto-Flag Triggers

After persisting coaching scores, backend evaluates:

```
flag_level = 'red'    if overall * 10 ≤ 5
                       OR objection_handling ≤ 1
                       OR (demo_discipline = 'generic' AND discovery_quality = 'abrupt')

flag_level = 'yellow' if overall * 10 BETWEEN 5.01 AND 6.5
                       OR any single phase_score < 0.40

flag_level = 'green'  otherwise
```

Flag is written to `bd_coaching_scores.flag_level` AND surfaced to BD Lead's coaching dashboard.

## Coaching SLA

| Flag | SLA |
|---|---|
| **Red** | BD Lead MUST schedule a 1:1 coaching session within **48 hours**. Backend creates a Calendar invite + sends Telegram nudge at 24h if no session booked. |
| **Yellow** | Address in next scheduled 1:1 (no separate session required). Surfaced on Lead's dashboard with yellow banner. |
| **Green** | No action; positive reinforcement Telegram to BD ("Solid call! Phase X scored 0.92 — keep going."). |

Cron `coaching_sla_enforcer` runs hourly, checks red-flag rows where `coaching_session_scheduled_at IS NULL` AND `flagged_at < NOW() - INTERVAL '24 hours'` → escalate to BD Lead's lead.

## DB Schema

```sql
CREATE TYPE coaching_flag_level AS ENUM ('red', 'yellow', 'green');
CREATE TYPE narrative_status AS ENUM ('generated', 'invalid_format', 'failed');

CREATE TABLE bd_coaching_scores (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id          UUID NOT NULL REFERENCES workspaces(id),
  prospect_id           UUID NOT NULL,
  extraction_run_id     UUID NOT NULL REFERENCES extractions(id),
  bd_email              VARCHAR(255) NOT NULL,
  call_date             DATE NOT NULL,
  call_phase_scores     JSONB NOT NULL,            -- {opening: 0.85, discovery_pain: 0.60, ...}
  question_scores       JSONB NOT NULL,            -- array of {question_id, score, evidence_quote}
  -- 5-dim quality score
  sop_score             DECIMAL(3,2) NOT NULL,
  discovery_score       DECIMAL(3,2) NOT NULL,
  feature_gap_score     DECIMAL(3,2) NOT NULL,
  objection_score       DECIMAL(3,2) NOT NULL,
  demo_score            DECIMAL(3,2) NOT NULL,
  overall_quality       DECIMAL(3,2) NOT NULL,     -- the weighted sum
  -- Flagging
  flag_level            coaching_flag_level NOT NULL,
  flagged_at            TIMESTAMPTZ,                -- non-null for red/yellow
  coaching_session_scheduled_at TIMESTAMPTZ,        -- set when Lead books session
  coaching_session_completed_at TIMESTAMPTZ,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(prospect_id, call_date)
);

CREATE INDEX idx_bcs_workspace_bd ON bd_coaching_scores(workspace_id, bd_email, call_date DESC);
CREATE INDEX idx_bcs_red_pending ON bd_coaching_scores(workspace_id, flagged_at)
  WHERE flag_level = 'red' AND coaching_session_scheduled_at IS NULL;

CREATE TABLE bd_deal_scores (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id          UUID NOT NULL REFERENCES workspaces(id),
  prospect_id           UUID NOT NULL,
  extraction_run_id     UUID NOT NULL REFERENCES extractions(id),
  call_day              VARCHAR(10) NOT NULL,      -- 'D0' | 'D2' | 'D7' | 'D14' | 'D21'
  next_step             DECIMAL(3,2) NOT NULL,
  pain_quantified       DECIMAL(3,2) NOT NULL,
  dm_identified         DECIMAL(3,2) NOT NULL,
  urgency               DECIMAL(3,2) NOT NULL,
  product_fit           DECIMAL(3,2) NOT NULL,
  overall_deal          DECIMAL(3,2) NOT NULL,
  computed_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(prospect_id, call_day)
);

CREATE INDEX idx_bds_workspace_day ON bd_deal_scores(workspace_id, call_day, overall_deal);

CREATE TABLE bd_coaching_narratives (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id          UUID NOT NULL REFERENCES workspaces(id),
  prospect_id           UUID NOT NULL,
  coaching_score_id     UUID NOT NULL REFERENCES bd_coaching_scores(id),
  bd_email              VARCHAR(255) NOT NULL,
  narrative_text        TEXT NOT NULL,             -- 250-400 words Indonesian
  narrative_status      narrative_status NOT NULL,
  word_count            INTEGER NOT NULL,
  delivered_telegram_at TIMESTAMPTZ,
  delivered_email_at    TIMESTAMPTZ,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_bcn_workspace_bd ON bd_coaching_narratives(workspace_id, bd_email, created_at DESC);
CREATE INDEX idx_bcn_pending_delivery ON bd_coaching_narratives(workspace_id, created_at)
  WHERE delivered_telegram_at IS NULL OR delivered_email_at IS NULL;
```

## Distribution

Cron `coaching_distribution` runs daily at 08:00 WIB. Sends every coaching narrative generated since 08:00 WIB previous day to its BD owner via:

1. **Telegram DM** — `narrative_text` truncated to 4096 chars, with link to full narrative in dashboard
2. **Email** — full narrative + link to Prospect Detail Drawer + link to call recording

```sql
SELECT id, workspace_id, bd_email, narrative_text, prospect_id
FROM bd_coaching_narratives
WHERE workspace_id = $1
  AND narrative_status = 'generated'
  AND (delivered_telegram_at IS NULL OR delivered_email_at IS NULL)
  AND created_at > NOW() - INTERVAL '36 hours';
```

On successful send → set `delivered_telegram_at` / `delivered_email_at`. On failure → retry exponential per `04-integrations.md` retry policy.

If BD has 3+ red-flag narratives in a 7-day window, the 08:00 WIB distribution also CC's the BD Lead with a summary.

## API Surface

### GET `/coaching/scores`

Returns paginated scores for the active workspace. BD users see only own; Lead/Admin see all.

```
Query: ?bd_email=alice@dealls.com&flag_level=red&from_date=2026-04-01&page=1&per_page=20

Response 200:
{
  "data": [
    {
      "id": "uuid-xxx",
      "prospect_id": "uuid-yyy",
      "company_name": "PT Maju Jaya",
      "bd_email": "alice@dealls.com",
      "call_date": "2026-04-21",
      "overall_quality": 0.42,
      "flag_level": "red",
      "phase_scores": { "opening": 0.75, "discovery_pain": 0.30, ... }
    }
  ],
  "total": 8
}
```

### GET `/coaching/narratives/{score_id}`

Full narrative + scores combined.

### POST `/coaching/sessions`

Lead books a coaching session for a red-flagged score. Sets `coaching_session_scheduled_at`.

```
Request:
{
  "coaching_score_id": "uuid-xxx",
  "scheduled_at": "2026-04-23T10:00:00+07:00",
  "session_url": "https://meet.google.com/xxx"
}

Response 200: { "session_id": "uuid-zzz", "scheduled_at": "..." }
```

### POST `/coaching/sessions/{id}/complete`

Lead marks session complete after 1:1. Optional `coach_notes` field.

## Backend MUST

- Persist coaching scores in the SAME transaction as the extraction's Stage 3 output (atomic — both succeed or neither)
- Compute `overall_quality` and `overall_deal` server-side from the stored sub-components (do NOT trust Claude's claimed `overall`)
- Validate narrative contains all 4 section headers; one retry on invalid format
- Apply auto-flag rules immediately after score persist
- Schedule coaching distribution for next 08:00 WIB delivery
- Trigger `ESC-BD-003` at D7 when D2 deal_score < 0.5

## Backend MAY

- Aggregate weekly coaching trends per BD into a `bd_coaching_weekly` materialised view for Lead dashboard
- Auto-suggest coaching topics based on the most-failed phase across a BD's last 10 calls (computed offline)
- Allow Lead to mark a score as `manually_overridden` with a justification (audit-logged)
